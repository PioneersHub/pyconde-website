# Redirects

Two redirect pipelines run side by side, both feeding the same static-HTML + nginx + Caddy outputs:

1. **Manual redirects** — declared in `databags/redirects.yaml` for arbitrary `/old/` → `/new/` mappings (renamed pages, retired URLs, vanity short-links).
2. **Pretalx-code redirects** — generated by the slug migration, mapping `/archive/{year}/talks/{CODE}/` → `/archive/{year}/talks/{slug}/`. Auto-maintained.

Both produce static HTML pages with `<meta http-equiv="refresh">`, a JS `window.location.replace`, a canonical link, and `robots: noindex,follow`. The HTML pages work the instant the build deploys — no nginx config rollout needed. The nginx and Caddy snippets are emitted for the hoster to wire up real `301`s on the next deploy window; once those are in place the HTML pages become redundant but harmless.

## Manual redirects

Add an entry to `databags/redirects.yaml`:

```yaml
redirects:
  - from: /pyladies-at-pyconde-2025/
    to:   /blog/pyladies/
    note: 2025 PyLadies page consolidated into blog series
  - from: /tutorials/
    to:   /masterclasses/
    status: 301
    note: Tutorials renamed to Masterclasses for 2026
```

`from` must start and end with `/`. `to` is either an absolute path or a full URL. `status` is informational (HTML redirects always behave like a 301 once cached). `note` shows up as a comment in the generated server snippets.

Generate:

```bash
make redirects
```

Which writes:

| Output | What |
|---|---|
| `content/pyladies-at-pyconde-2025/contents.lr` (etc.) | One Lektor `_model: redirect` page per entry; rendered into a static HTML redirect by `make build` |
| `site-config/redirects.nginx.conf` | `location = /old/ { return 301 /new/; }` lines for the hoster (gitignored) |
| `site-config/Caddyfile.redirects` | `redir /old/ /new/ 301` lines for staging (gitignored) |

The generator carries a marker comment on every page it writes; removing an entry from the YAML and re-running prunes the corresponding `content/` folder automatically. Hand-authored pages without the marker are never touched.

## Pretalx-code redirects

Run after every archive cutover or slug change:

```bash
uv run python utils/migrate_pretalx_slugs.py
```

The script (idempotent):

1. Walks every `/talks/` directory under `content/` and `content/archive/{year}/`.
2. For each talk folder still named by its Pretalx code, slugifies the title (or reuses the existing `slug:` field), renames the folder, and writes the new slug into `contents.lr`.
3. Updates each speaker's `talks:` field to reference talks by slug.
4. Creates a sibling content folder named by the Pretalx code with `_model: redirect` and `target: /…/talks/{slug}/`.
5. Appends server-side snippets to `site-config/pretalx-redirects.nginx.conf` and `site-config/Caddyfile.pretalx-redirects`.

Collisions (two talks slugifying to the same string) are resolved deterministically by suffixing the losing entry with `-{code}` — predictable, stable across re-runs.

A talk whose Pretalx code happens to match its slug (rare) skips the redirect step — there's nothing to redirect to.

## Operational notes

- **`make build` runs `make redirects` first** so the YAML-driven redirects are always in sync with the latest YAML before Lektor builds. The Pretalx-code redirects are not run on every build (they only need to run after a content rename); call `migrate_pretalx_slugs.py` explicitly when needed.
- **Don't delete a `from:` whose source still has live content.** Remove the source page first, otherwise both the original and the redirect compete for the URL slot.
- **Don't redirect into a 404.** The generator does not validate `to:` targets. Use the local static server (`make build` + `python -m http.server --directory site`) to spot-check.
- **Server-side 301s win once deployed.** Browsers and crawlers prefer the HTTP-level redirect over the HTML one; the HTML page only fires if the request reaches Lektor's output untouched by nginx/Caddy. Both keep working in parallel.
